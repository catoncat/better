# MES 产品交互 Review — Meta 任务清单

> 状态：草案（待确认）
> 更新时间：2026-01-28
> 目标：建立“可持续的交互/体验 review 机制”，用于梳理 MES 前端在功能膨胀后的混乱状态，并将改进沉淀为可追踪产出。

---

## 0. 背景与问题（Why）

当前 MES 前端页面多数由 AI 在不同功能点“拼接式生成”，随着功能膨胀出现：
- 业务流程不闭环/门禁不一致（用户会卡死或走错路）
- 功能入口与模块边界混乱（同类能力分散、重复、难发现）
- UX 反馈、错误处理、空态不统一（不可预测）
- UI 风格与组件用法不一致（维护成本高）

需要把“整理”从一次性大改，转为可持续的 review 机制。

---

## 1. 评审优先级（不可颠倒）

业务逻辑通顺 > 功能规划合理 > UX > UI

说明：
- 先保证流程可闭环、门禁正确、状态语义一致；
- 再调整功能规划/信息架构（入口、分组、职责边界）；
- 再做体验层优化；
- UI 统一放在最后（否则容易“粉饰太平”）。

---

## 2. 真源与证据（裁决顺序）

1. 流程/状态机：`domain_docs/mes/spec/`
2. API 合约/概览：`domain_docs/mes/tech/api/`
3. 权限与展示策略：`domain_docs/mes/permission_audit.md`
4. 前端实现：`apps/web/src/routes/_authenticated/mes/`
5. 演示/用户文档：`user_docs/`

---

## 3. 交付物结构（持久化真源）

统一落在：`domain_docs/mes/ux_review/`

- `00_review_method.md`：方法与准则（优先级、证据规范、问题分类）
- `00_interaction_matrix_template.md`：交互矩阵模板（用于 round 产出）
- `00_review_backlog.md`：Review 范围清单（按“用户旅程/核心页面”拆分）
- `00_status.md`：共享进度状态（轮次/高风险项追踪）
- `roundN_*.md`：每轮执行产出（交互矩阵 + 问题清单 + 建议）
- `99_high_risk_findings.md`：高风险误导点汇总（可选）

---

## 4. Meta 任务清单

### T-UX-0 方法与模板
- T-UX-0.1 定义 Review 方法文档（优先级、证据、问题分类、输出规范）
- T-UX-0.2 定义交互矩阵模板（用户意图 ↔ UI ↔ API ↔ 状态 ↔ 权限 ↔ 异常）

### T-UX-1 Backlog（范围入口）
- T-UX-1.1 从核心流程/权限审计/路由清单抽取 review 范围（Backlog）
- T-UX-1.2 定义轮次策略（不锁死顺序，但给出推荐分组）

### T-UX-2 轮次执行（由 ux-review-exec 承接）
- T-UX-2.x 每轮产出一个 `roundN_*.md`（矩阵 + 问题清单 + 建议）
- T-UX-2.y 高风险误导点汇总（可选）

### T-UX-3 从发现到改进（承接到开发）
- T-UX-3.1 形成“可执行的改进 Backlog”（按 P0/P1/P2）
- T-UX-3.2 由 `/next` 或 `/dev` 把改进项拆进开发计划并落地
